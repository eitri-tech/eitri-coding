# SKILL.md — Claude Design → Eitri Migration

## When to use this skill

Trigger automatically when the user asks for any of:

- "migrate this Claude Design code to Eitri"
- "adapt these screens to Luminus / eitri-luminus"
- "port this design to our Eitri app"
- "turn this HTML/JSX from Claude Design into Eitri views"
- Any request that points at JSX/HTML containing `DesignCanvas`, `DCSection`, `DCArtboard`, `IOSDevice`, `AndroidDevice`, screens built from `<div style={{...}}>` trees with palette/font constants, or a Babel-standalone HTML host file.

The source can be **anywhere** — a folder the user points at, a single `.jsx` / `.html` file, or even a snippet pasted directly in the conversation. Do not assume any fixed path. If the user just pastes code, treat that snippet as the source and write the migrated output where the user asks (or, if unspecified, ask once where to place it).

If unsure whether the input is Claude Design output, ask the user once: *"Is this Claude Design output you want migrated to an Eitri view?"* — if yes, run this skill.

---

## Goal

Produce idiomatic Eitri (Luminus) source files that:

1. Reproduce the **layout, hierarchy, spacing, colors and typography** of the source design.
2. Use **only `eitri-luminus` components** — no raw HTML tags.
3. Use **TailwindCSS + DaisyUI v4** classes where possible, falling back to component props (`width`, `height`, `padding`, `backgroundColor`, etc.) when a Tailwind utility does not map cleanly.
4. Drop everything that is **device chrome / preview scaffolding** (status bars, notches, home indicator, `DesignCanvas`, `DCArtboard`, `IOSDevice`, `AndroidDevice` wrappers). These exist only to render the design in a browser and have no place inside an Eitri app.
5. Follow Eitri file-based routing under `src/views/`.

---

## Mandatory rules (inherited from `eitri-coding`)

- **No HTML tags.** `div`, `span`, `p`, `img`, `button`, `input`, `a`, `h1`–`h6`, `ul`, `li`, `section`, `header`, `footer`, `nav`, `svg` *containers*, etc. are forbidden. Always use `eitri-luminus` equivalents.
- **No `hover:`, `focus:`, `focus-within:`, `active:` Tailwind utilities** — they get stuck on mobile touchscreens.
- **Sizing as props:** `width`, `height`, `maxWidth`, `maxHeight`, `minWidth`, `minHeight` are direct props on Luminus components.
- **Default orientation is `row`.** Add `orientation="column"` (or `column` shorthand if available) when a vertical stack is needed.
- **Mobile-only layouts.** Eitri apps are mobile mini-apps — no responsive desktop breakpoints.
- **Function components only:** `export default function ScreenName(props) { ... }` — no arrow functions for the main export, no destructuring in the signature.
- **Pull params from props:**
  - URL: `const { id } = props.match.params`
  - State: `const { data } = props.location.state`

## Mapping cheat-sheet (Claude Design → Luminus)

Use this as the default substitution table. When in doubt about a prop or an exact component name, consult the official component list with `WebFetch`:

> https://cdn.83io.com.br/library/luminus-ui/doc/latest/components/

| Claude Design source                                  | Eitri Luminus replacement                                                              |
| ----------------------------------------------------- | -------------------------------------------------------------------------------------- |
| `<div style={{ display: 'flex', flexDirection: 'column' }}>` | `<View orientation="column">`                                                  |
| `<div style={{ display: 'flex' }}>` (default row)     | `<View>`                                                                               |
| `<div>` acting as a card / box                        | `<View>` with Tailwind `className="bg-base-100 rounded-2xl p-4"` (or props)            |
| `<span>` / `<p>` / `<h1>`–`<h6>` text nodes           | `<Text>` (use `className` for typography, e.g. `text-2xl font-semibold`)               |
| `<button>`                                            | `<Button>` (Luminus) — never raw `<button>`                                            |
| `<input>` / `<textarea>`                              | `<Input>` / `<Textarea>` from Luminus                                                  |
| `<img src=...>`                                       | `<Image src=... />` from Luminus                                                       |
| Inline `<svg>` icons                                  | Keep the SVG **inside** an Eitri-compatible icon wrapper (typically `react-icons` from the supported version, or wrap in `<View>` containing the `<svg>` if it's purely decorative inside a component that renders raw markup). Prefer `react-icons` when an equivalent exists. |
| `style={{ background: '#fff' }}`                      | `backgroundColor="#fff"` prop or Tailwind `bg-[#fff]`                                  |
| `style={{ padding: 16 }}`                             | `padding={16}` prop or Tailwind `p-4`                                                  |
| `style={{ gap: 8 }}`                                  | `gap={8}` prop or Tailwind `gap-2`                                                     |
| `boxShadow`, `borderRadius`                           | Tailwind `shadow-*`, `rounded-*` (or use arbitrary values: `rounded-[14px]`)           |
| `fontFamily: '"Inter", ...'`                          | Do **not** hard-code in the component. Configured globally via the Eitri Developer Console (see *Fonts* below). |
| Device chrome (`IOSDevice`, `AndroidDevice`, status bars, notches, home indicators) | **Delete.** Never migrated.                                  |
| Canvas (`DesignCanvas`, `DCSection`, `DCArtboard`)    | **Delete.** Each `DCArtboard` becomes a separate file under `src/views/...`.           |

### Color tokens & Tailwind theme

If the source defines a palette object (e.g. `const LOOP = { blue: '#1E5BFF', ... }`):

- For a one-off color, inline it via a Tailwind arbitrary class (`bg-[#1E5BFF]`) — fine.
- For **any palette that is reused** (brand colors, semantic states, neutrals, custom font families), do **not** edit a local `tailwind.config.js`. Eitri Luminus v2 reads its Tailwind / DaisyUI theme from a JSON configured at **Eitri Console → Design System → Luminus V2.x**, and every Eitri-App on Luminus v2 inherits it. You must hand the user a complete, ready-to-paste JSON with the design's tokens mapped onto the **default theme name `eitri`**.

#### How to deliver the theme JSON

When the migrated design needs custom colors and/or fonts, output a single JSON block following this exact structure (this is the format the Eitri Developer Console expects):

- `daisyui.themes[0].eitri` — the main DaisyUI theme. The theme key **must be `eitri`** and `defaultTheme` **must be `"eitri"`**, since this is the name every Luminus v2 app inherits.
- Always include the full DaisyUI semantic set: `primary`, `primary-content`, `secondary`, `secondary-content`, `accent`, `accent-content`, `neutral`, `neutral-content`, `base-100`, `base-200`, `base-300`, `base-content`, `info`, `info-content`, `success`, `success-content`, `warning`, `warning-content`, `error`, `error-content`. Map the design's palette onto these slots — pick sensible defaults for any slot the design didn't explicitly define (don't leave a slot out).
- `theme.extend.colors` — extra named colors from the design that don't fit DaisyUI semantics (custom neutrals, brand-specific shades, etc.). Use Tailwind-style names (`neutral-500`, `brand-blue`, `yellow-800`, …).
- `theme.extend.fontFamily` — the fonts used in the design, in Tailwind format. These pair with the Google Fonts `@import` shipped via the Theme CSS (see *Final hand-off*).
- `darkTheme: false` unless the design is explicitly dark.

Reference shape (this is the literal format shown in the Eitri Developer Console — keep it 1:1):

```json
{

    "themes": [
      {

          "primary": "#FF77AA",

          "accent-content": "#000000",
          "neutral": "#fafafa",
          "neutral-content": "#000000",
          "base-100": "#FEFCFB",
          "base-200": "#35323e",
          "base-300": "#484554",
          "base-content": "#e4e3e8",
          "info": "#306ee8",
          "info-content": "#ffffff",
          "success": "#22c322",
          "success-content": "#000000",
          "warning": "#eebd2b",
          "warning-content": "#000000",
          "error": "#df2020",
          "error-content": "#ffffff"
        }
      }
    ],
    "defaultTheme": "eitri",
    "darkTheme": false
```

After producing the JSON, tell the user:

> Open **Eitri Console → Design System → Luminus V2.x** and paste this JSON. The theme is named `eitri` and is set as default, so every Luminus v2 app in your workspace will inherit it automatically. Use `bg-primary`, `text-primary-content`, `bg-base-100`, etc. in the migrated components instead of inline hex values.

Once the theme JSON is in place, **prefer DaisyUI semantic classes** (`bg-primary`, `text-base-content`, `bg-base-200`) over arbitrary hex classes inside the migrated components — that way a future re-skin only changes the JSON, not the code.

## Migration workflow

For every migration request, follow these steps in order. Use `Read`, `Grep`, `Glob` and `Agent` (with `subagent_type=Explore`) as needed to understand the source.

### 1. Inventory the source

- Locate the input: a folder, a file, or an inline snippet. Don't assume a fixed path — use whatever the user provided.
- Read every screen / artboard found there.
- List the **screens** to migrate (one Luminus file each) and the **shared primitives** (buttons, fields, progress dots, marks, etc.) — those become reusable components, ideally under `src/components/`.
- Identify the **palette** and **fonts** in use. Note them — you'll surface them at the end.
- If the input is a single snippet with no obvious destination, ask the user once where the migrated files should live before writing anything.

### 2. Strip the scaffolding

Identify and discard everything that is preview-only:

- `DesignCanvas`, `DCSection`, `DCArtboard` wrappers.
- `IOSDevice`, `AndroidDevice`, status bars (time, battery, signal), notches, dynamic islands, home indicators, gesture bars.
- The HTML host file (`<!doctype html>`, Babel standalone scripts, font `<link>` tags).

The migrated output contains **only the inner screen content**.

### 3. Plan the file layout

Default mapping (adapt to the user's existing project structure if one exists):

```
src/
  views/
    Onboarding/
      Welcome.tsx
      Phone.tsx
      Verify.tsx
      Profile.tsx
      Plan.tsx
      Done.tsx
  components/
    PrimaryButton.tsx
    GhostButton.tsx
    TextField.tsx
    ProgressDots.tsx
    BrandMark.tsx
```

Confirm this with the user **only** if the project already has a different convention; otherwise proceed.

### 4. Convert each file

For every screen / primitive:

1. Replace HTML tags with the Luminus equivalents from the cheat-sheet.
2. Replace `style={{...}}` with a combination of Luminus props (`width`, `height`, `padding`, `gap`, `backgroundColor`, `orientation`) and Tailwind / DaisyUI classes.
3. Remove forbidden Tailwind pseudo-utilities (`hover:`, `focus:`, `focus-within:`, `active:`).
4. Do not destructure props in the signature; pull them off `props` inside the body.
5. Default orientation is `row` — add `orientation="column"` when the source was a vertical flex column.
6. Keep the **layout intent**: spacing, alignment, sizes, ordering. Pixel-perfect parity is the bar.
7. Strip ad-hoc animations (`@keyframes caret`, etc.) unless the user asks for them — Eitri styling is centralized.

### 5. Use `WebFetch` when uncertain

If you don't know whether a Luminus component exists or what its prop is:

```
WebFetch https://cdn.83io.com.br/library/luminus-ui/doc/latest/components/
```

Never guess component names.

### 6. Final hand-off — fonts & theme

If the source design uses **specific fonts** (Inter, Fragment Mono, EB Garamond, etc.), do **not** import them in the migrated component code. Instead, at the end of the migration, instruct the user to paste the font CSS at **Eitri Console → Design System → Luminus V1.x** (the V1.x section is where the Theme CSS / `@import` font block lives — even for projects on Luminus v2, fonts go through V1.x). Reference:

> https://docs.eitri.tech/en/concepts/design-system/#theme

Provide the user with a ready-to-paste CSS snippet, in this exact shape (adjust the families to match the actual fonts used in the design):

```css
@import url(https://fonts.googleapis.com/css2?family=EB+Garamond:ital,wght@0,400..800;1,400..800&family=Inter:ital,opsz,wght@0,14..32,100..900;1,14..32,100..900&display=swap);

body {
    font-family: Inter, sans-serif;
    background: #fff
}
```

Tell the user, in plain text:

> The migrated code does not import fonts directly. Open **Eitri Console → Design System → Luminus V1.x** and paste the CSS snippet above (already tailored to the fonts used in your design). The `body { font-family: ... }` rule will cascade through every Luminus component.

If the source used DaisyUI semantic colors / a custom theme, also output the full Tailwind / DaisyUI theme JSON described in *Color tokens & Tailwind theme* above (theme key `eitri`, set as default) and tell the user to paste it at **Eitri Console → Design System → Luminus V2.x**. So:

- **Fonts (CSS `@import` + `body`)** → Eitri Console → Design System → **Luminus V1.x**
- **Colors / Tailwind / DaisyUI JSON** → Eitri Console → Design System → **Luminus V2.x**

Both live in the Eitri Console — never edit a local `tailwind.config.js`.

## Output checklist

Before reporting the migration complete, verify:

- [ ] No raw HTML tags remain (`grep -nE '<(div|span|p|img|button|input|a|h[1-6]|ul|li|section|header|footer|nav)\b'` on the migrated files returns nothing).
- [ ] No forbidden Tailwind pseudo-classes (`grep -nE '\b(hover|focus|focus-within|active):' on the migrated files`).
- [ ] No `IOSDevice` / `AndroidDevice` / `DesignCanvas` / `DCArtboard` references.
- [ ] No inline `@font-face` / `<link rel="stylesheet">` for fonts.
- [ ] All screens are `export default function Name(props) { ... }`.
- [ ] Each screen is at `src/views/.../<Name>.tsx` (or `.jsx`, matching project convention).
- [ ] Shared primitives extracted to `src/components/`.
- [ ] User has been given the font / theme CSS snippet and pointed to the Eitri Developer Console.
- [ ] If the design has custom colors or fonts, the full Tailwind / DaisyUI theme JSON has been delivered with `defaultTheme: "eitri"` and the theme key named `eitri`.

## Tool usage

- **`Read` / `Grep` / `Glob`:** read every source screen and primitive before generating output.
- **`Agent` (`subagent_type=Explore`):** when the design spans many files, delegate the inventory pass.
- **`WebFetch`:** Luminus component reference and the Eitri design-system docs.
- **`Write` / `Edit`:** prefer `Edit` when a target file already exists; use `Write` to create new view / component files.
- **`Bash`:** only for `grep`-style verification of the output checklist and (optionally) running `eitri start` to preview the result.

---

## Mindset

- The source is a **design**, not a contract. Reproduce intent, not implementation details.
- Device frames are noise — always strip them.
- Centralize anything global (fonts, theme colors) in the Eitri Developer Console; keep components clean.
- Never invent Luminus components or props — verify against the official docs.
